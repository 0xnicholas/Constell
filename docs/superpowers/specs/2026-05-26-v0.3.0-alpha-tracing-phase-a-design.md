# v0.3.0-alpha Tracing — Phase A Design Spec

**Version:** 0.3.0-alpha-phase-a  
**Date:** 2026-05-26  
**Status:** Draft  
**Scope:** Ingestion Pipeline (Data Flow Closed Loop)

---

## 1. Goal

SDK / API client sends a trace → Ingestion API validates + enqueues → Worker processes batch → Writes to ClickHouse (immutable events) + PostgreSQL (metadata).

**Exit Criteria:** `curl` sends trace → worker writes → `SELECT * FROM observations_wide` returns data.

---

## 2. Architecture

```
Client (curl)
    │ POST /api/public/ingestion
    │ { batch: [ { type: "trace-create", body: {...} }, ... ] }
    ▼
┌─────────────────────────────────────────┐
│  web (Next.js)                          │
│  ├─ validateApiKey (PostgreSQL)         │
│  ├─ shallow validation (Zod)            │
│  ├─ enqueue to BullMQ                   │
│  └─ return 202 { batchId }              │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│  Redis (BullMQ ingestion-queue)         │
│  ├─ retry: 3 attempts                   │
│  ├─ backoff: exponential                │
│  └─ dead letter after max attempts      │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│  worker (Express + BullMQ Worker)       │
│  ├─ ingestionProcessor                  │
│  │  ├─ 1. Parse events from batch       │
│  │  ├─ 2. Deep validate each event      │
│  │  ├─ 3. Dedupe by event.id            │
│  │  ├─ 4. Enrich (cost calc, timestamps)│
│  │  ├─ 5. Write traces to PG            │
│  │  ├─ 6. Write observations to PG      │
│  │  ├─ 7. Flush traces to CH (batch)    │
│  │  ├─ 8. Flush observations to CH      │
│  │  └─ 9. Report failed events          │
│  └─ blobStorageProcessor (placeholder)  │
└─────────────────────────────────────────┘
    │
    ├──► PostgreSQL (metadata)
    │     Trace, Observation (upsert)
    │
    └──► ClickHouse (immutable events)
          traces_wide, observations_wide (batch insert)
```

## 3. Event Batch Schema

### 3.1 Request Format

```typescript
interface IngestionBatch {
  batch: IngestionEvent[];
}

interface IngestionEvent {
  id: string; // client-generated event ID for deduplication
  type: EventType;
  timestamp: string; // ISO 8601
  body: EventBody;
}

type EventType = "trace-create" | "observation-create" | "observation-update"; // v0.3.0: only create

type EventBody = TraceCreateBody | ObservationCreateBody;
```

### 3.2 Trace Create Body

```typescript
interface TraceCreateBody {
  id: string; // trace ID (user-provided or generated)
  name?: string;
  userId?: string;
  sessionId?: string;
  metadata?: Record<string, unknown>;
  release?: string;
  version?: string;
  tags?: string[];
  public?: boolean;
  timestamp?: string; // overrides event.timestamp
}
```

### 3.3 Observation Create Body

```typescript
interface ObservationCreateBody {
  id: string;
  traceId: string;
  type: "SPAN" | "GENERATION" | "EVENT";
  name?: string;
  startTime?: string; // ISO 8601
  endTime?: string; // ISO 8601
  model?: string; // required for GENERATION
  input?: unknown; // any JSON-serializable value
  output?: unknown;
  usage?: {
    input?: number;
    output?: number;
    total?: number;
    unit?: "TOKENS" | "CHARACTERS" | "MILLISECONDS";
  };
  modelParameters?: Record<string, unknown>;
  metadata?: Record<string, unknown>;
  parentObservationId?: string;
  level?: "DEBUG" | "DEFAULT" | "WARNING" | "ERROR";
  statusMessage?: string;
  environment?: string;
  release?: string;
  version?: string;
  sessionId?: string;
  userId?: string;
}
```

### 3.4 Event Size Limits

- Max batch size: 100 events
- Max event body size: 1MB
- Max total request size: 4MB

## 4. Ingestion API (Web)

### 4.1 Endpoint

```
POST /api/public/ingestion
Authorization: Basic <base64(pk:sk)>
Content-Type: application/json
```

### 4.2 Validation Pipeline

```
1. Parse Authorization header → validate API key
2. Check rate limit (v0.3.0-phase-b: placeholder)
3. Parse JSON body → check max size
4. Zod validate batch structure (shallow)
5. Enqueue to BullMQ
6. Return 202 Accepted
```

### 4.3 Response

```typescript
// 202 Accepted
{
  batchId: string;    // BullMQ job ID
  successes: number;  // number of events accepted
}

// 400 Bad Request
{
  error: "VALIDATION_ERROR",
  message: string;
  details?: z.ZodError;
}

// 401 Unauthorized
{
  error: "UNAUTHORIZED",
  message: string;
}

// 429 Too Many Requests
{
  error: "RATE_LIMITED",
  message: string;
  retryAfter: number;
}
```

## 5. Worker Ingestion Processor

### 5.1 Job Handler

```typescript
// worker/src/queues/ingestionProcessor.ts
export async function processIngestionJob(job: Job<IngestionJob>): Promise<IngestionResult> {
  const { batch, projectId } = job.data;

  const ctx: ProcessingContext = {
    projectId,
    jobId: job.id,
    timestamp: new Date(),
  };

  // Phase 1: Parse + shallow validate
  const events = parseEvents(batch);

  // Phase 2: Deep validate each event
  const { valid, invalid } = await validateEvents(events, ctx);

  // Phase 3: Dedupe by event.id (within batch)
  const deduped = dedupeEvents(valid);

  // Phase 4: Enrich (cost calculation, timestamp normalization)
  const enriched = await enrichEvents(deduped, ctx);

  // Phase 5: Categorize by type
  const traces = enriched.filter((e) => e.type === "trace-create");
  const observations = enriched.filter((e) => e.type === "observation-create");

  // Phase 6: PostgreSQL upserts (metadata)
  const pgResult = await writeToPostgres({ traces, observations }, ctx);

  // Phase 7: ClickHouse batch inserts (immutable events)
  const chResult = await writeToClickHouse({ traces, observations }, ctx);

  // Phase 8: Report failures
  return {
    processed: enriched.length,
    failed: invalid.length + pgResult.failed + chResult.failed,
    failures: [...invalid, ...pgResult.failures, ...chResult.failures],
  };
}
```

### 5.2 Deep Validation Rules

| Event Type           | Validation Rules                                                                                                                         |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `trace-create`       | `body.id` required, `name` optional string, `metadata` optional object                                                                   |
| `observation-create` | `body.id` required, `body.traceId` required (must exist in PG or same batch), `body.type` enum, `body.model` required if type=GENERATION |

**Cross-reference validation:**

- `observation-create.traceId` must exist in PostgreSQL `Trace` table OR be present in a `trace-create` event within the same batch.

### 5.3 Enrichment

| Field            | Enrichment Logic                                                                                                |
| ---------------- | --------------------------------------------------------------------------------------------------------------- |
| `event_ts`       | Use `event.timestamp` or `body.timestamp` or `now()`                                                            |
| `latency_ms`     | `(endTime - startTime)` in ms, or null                                                                          |
| `calculatedCost` | Lookup `ModelPrice` by `model`, calculate `(inputTokens * inputPrice + outputTokens * outputPrice) / 1_000_000` |
| `totalTokens`    | `usage.total` or `usage.input + usage.output`                                                                   |
| `level`          | Default to "DEFAULT" if not provided                                                                            |

### 5.4 PostgreSQL Writes (Idempotent)

```sql
-- Trace: ON CONFLICT for (projectId, externalId)
INSERT INTO "Trace" (id, externalId, projectId, name, ...)
VALUES (...)
ON CONFLICT ("projectId", "externalId") DO NOTHING;

-- Observation: ON CONFLICT for id
INSERT INTO "Observation" (id, traceId, projectId, type, ...)
VALUES (...)
ON CONFLICT (id) DO NOTHING;
```

### 5.5 ClickHouse Writes (Batch)

```typescript
// Accumulate rows in memory
const chTraces: ClickHouseTraceRow[] = [];
const chObservations: ClickHouseObservationRow[] = [];

// Flush when buffer reaches 1000 rows OR job ends
async function flushClickHouse(): Promise<void> {
  if (chTraces.length > 0) {
    await clickhouse.insert({
      table: "traces_wide",
      values: chTraces,
      format: "JSONEachRow",
    });
    chTraces.length = 0;
  }
  if (chObservations.length > 0) {
    await clickhouse.insert({
      table: "observations_wide",
      values: chObservations,
      format: "JSONEachRow",
    });
    chObservations.length = 0;
  }
}
```

### 5.6 Failure Handling

```typescript
interface ProcessingFailure {
  eventId: string;
  eventType: EventType;
  reason: string; // human-readable
  code: string; // machine-readable: VALIDATION_ERROR, PG_ERROR, CH_ERROR
}

// Failed events are:
// 1. Logged to console (with job ID for correlation)
// 2. Returned in job result (for BullMQ job metadata)
// 3. v0.4.0: Written to dead-letter queue / table
```

**Failure Categories:**

| Code               | Cause                                         | Action                                                 |
| ------------------ | --------------------------------------------- | ------------------------------------------------------ |
| `VALIDATION_ERROR` | Schema mismatch, missing required field       | Skip event, report failure                             |
| `TRACE_NOT_FOUND`  | observation-create references unknown traceId | Skip event, report failure                             |
| `PG_ERROR`         | Prisma query failed                           | Skip event, report failure                             |
| `CH_ERROR`         | ClickHouse insert failed                      | Skip event, report failure (CH retries at batch level) |

## 8. Extension Points

| Future Feature       | How to Extend                                                                  |
| -------------------- | ------------------------------------------------------------------------------ |
| `observation-update` | Add new event type, update PG row + insert new CH row with newer `event_ts`    |
| `score-create`       | Add new event type, new CH table `scores`                                      |
| Batch export to S3   | Hook into `ingestionProcessor` post-processing, enqueue to `blobstorage-queue` |
| Real-time alerts     | Subscribe to BullMQ job completion events, trigger webhook                     |

## 9. Decision Log

| Decision                                | Context                                                      | Status    |
| --------------------------------------- | ------------------------------------------------------------ | --------- |
| Langfuse-style event batch              | Proven pattern, supports all future event types              | ✅ Frozen |
| Skip-fail (not all-or-nothing)          | Production reliability: one bad event shouldn't block others | ✅ Frozen |
| Sequential processing + CH batch buffer | Correctness-first, CH batching for throughput                | ✅ Frozen |
| ON CONFLICT DO NOTHING for PG           | Idempotent writes, client can retry safely                   | ✅ Frozen |
| CH `event_ts` as version column         | ReplacingMergeTree deduplication on re-ingestion             | ✅ Frozen |
