# v0.5.0-beta — Metrics & Analytics Design Spec

> **Status**: Draft  
> **Target Release**: v0.5.0-beta  
> **Depends on**: v0.4.0-alpha (Prompt Management)

---

## 1. Goal

Turn the raw trace/observation stream into actionable metrics dashboards and exportable reports. Users should be able to see cost, latency, token usage, and error trends in the UI, and export data to S3/MinIO for downstream analysis.

**Exit**: Dashboard shows accurate total cost, P95 latency, token usage trends, and model breakdown. Export pipeline generates JSONL/CSV to MinIO.

---

## 2. Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                          Web (Next.js)                           │
│  ┌─────────────┐  ┌─────────────┐                                │
│  │ Dashboard   │  │ tRPC        │                                │
│  │ Page        │◄─┤ metrics     │                                │
│  │             │  │ router      │                                │
│  └─────────────┘  └──────┬──────┘                                │
└──────────────────────────┼───────────────────────────────────────┘
                           │
                           ▼
              ┌────────────────────┐    ┌─────────────────┐
              │ ClickHouse         │    │ Redis (BullMQ)  │
              │ traces_wide FINAL  │    │ export-queue    │
              │ observations_wide  │    │ (new)           │
              └────────────────────┘    └────────┬────────┘
                                                 │
                                                 ▼
                                        ┌─────────────────┐
                                        │ Worker          │
                                        │ exportProcessor │
                                        │ (new consumer)  │
                                        └────────┬────────┘
                                                 │
                                                 ▼
                                        ┌─────────────────┐
                                        │ MinIO / S3      │
                                        │ .jsonl / .csv   │
                                        └─────────────────┘
```

**Design Principles**:

- **Query-time aggregation** over materialized views. CH `FINAL` + time-bounded scans are fast enough at beta scale; materialized views add operational burden.
- **Single metrics router** serves both UI and API consumers. Export is triggered via `metrics.export` tRPC mutation only (no separate REST endpoint in v0.5.0-beta).
- **Export is async via dedicated queue**. A separate `export-queue` (not `blobstorage-queue`) handles export jobs to avoid type collision with the existing `BlobStorageJob` contract.
- **`blobstorage-queue` remains a placeholder**. It is not repurposed for export; the existing placeholder worker continues to log and return `{ processed: true }`.
- **No server-side cache for beta**. Dashboard queries hit ClickHouse directly; query-time aggregation on `project_id` + time range is fast enough at this scale. Redis caching is listed as a post-v0.5.0 optimization.

## 3. Data Model & Query Design

### 3.1 ClickHouse Tables

Existing tables already carry the required fields:

**`traces_wide`**
| Field | Type | Notes |
|-------|------|-------|
| `project_id` | String | Filter |
| `total_tokens` | UInt32 | Sum |
| `total_cost` | Decimal64(12) | Sum |
| `latency_ms` | UInt32 | Avg, P95 |
| `observation_count` | UInt32 | Sum |
| `has_error` | UInt8 | Sum → error rate |
| `created_at` | DateTime64(3) | Time range, grouping |

**`observations_wide`**
| Field | Type | Notes |
|-------|------|-------|
| `project_id` | String | Filter |
| `type` | String | Filter (GENERATION) |
| `model` | String | Group-by |
| `input_tokens` | UInt32 | Sum |
| `output_tokens` | UInt32 | Sum |
| `calculated_cost` | Decimal64(12) | Sum |
| `level` | String | ERROR count |
| `start_time` | DateTime64(3) | Time range |

### 3.2 Query Patterns

#### Summary (KPI cards)

```sql
SELECT
  count() AS trace_count,
  sum(total_tokens) AS total_tokens,
  sum(total_cost) AS total_cost,
  avg(latency_ms) AS avg_latency,
  quantile(0.95)(latency_ms) AS p95_latency,
  sum(has_error) AS error_count,
  round(sum(has_error) / count() * 100, 2) AS error_rate
FROM traces_wide FINAL
WHERE project_id = {projectId: String}
  AND created_at >= {from: String}
  AND created_at <= {to: String}
```

> **Note:** Alias `error_count` cannot be referenced in the same SELECT level; use `sum(has_error) / count()` directly. When `count() == 0` (no data in range), the API layer must coerce `errorRate` to `0` (not `NaN` or `null`). All KPI fields default to `0` for empty ranges.

#### Trends (time series)

```sql
SELECT
  toStartOfDay(created_at) AS day,
  count() AS trace_count,
  sum(total_tokens) AS tokens,
  sum(total_cost) AS cost,
  avg(latency_ms) AS avg_latency,
  quantile(0.95)(latency_ms) AS p95_latency
FROM traces_wide FINAL
WHERE project_id = {projectId: String}
  AND created_at >= {from: String}
  AND created_at <= {to: String}
GROUP BY day
ORDER BY day
```

Granularity switch:

- `hour`: `toStartOfHour(created_at)`
- `day`: `toStartOfDay(created_at)`

#### Model Breakdown

```sql
SELECT
  model,
  count() AS generation_count,
  sum(input_tokens) AS input_tokens,
  sum(output_tokens) AS output_tokens,
  sum(calculated_cost) AS cost
FROM observations_wide FINAL
WHERE project_id = {projectId: String}
  AND type = 'GENERATION'
  AND start_time >= {from: String}
  AND start_time <= {to: String}
  AND model IS NOT NULL
  AND model != ''
GROUP BY model
ORDER BY cost DESC
```

> **Note:** Filters out rows with empty or null `model` so they don't appear as an unnamed group in the UI.

#### Export Query (traces)

```sql
SELECT *
FROM traces_wide FINAL
WHERE project_id = {projectId: String}
  AND created_at >= {from: String}
  AND created_at <= {to: String}
ORDER BY created_at
```

#### Export Query (observations)

```sql
SELECT *
FROM observations_wide FINAL
WHERE project_id = {projectId: String}
  AND start_time >= {from: String}
  AND start_time <= {to: String}
ORDER BY start_time
```

## 4. API Design

### 4.1 tRPC `metrics` Router

```typescript
// web/src/server/api/routers/metrics.ts

metrics.summary
  Input: { projectId?: string, from?: string, to?: string }
  // Defaults: from = 7 days ago, to = now (ISO strings)
  Output: {
    traceCount: number;
    totalTokens: number;
    totalCost: number;
    avgLatencyMs: number;
    p95LatencyMs: number;
    errorCount: number;
    errorRate: number;
  }

metrics.trends
  Input: { projectId?: string, from?: string, to?: string, granularity?: 'hour' | 'day' }
  // Defaults: from = 7 days ago, to = now. granularity = auto-selected.
  // Auto-selection: if explicit granularity is omitted, router picks 'hour' for ranges <= 48h,
  // 'day' otherwise. If explicit granularity is provided, router uses it as-is and returns
  // up to 90 buckets (no auto-switching). UI handles overflow with horizontal scroll.
  Output: {
    data: Array<{
      bucket: string;  // ISO date string
      traceCount: number;
      tokens: number;
      cost: number;
      avgLatencyMs: number;
      p95LatencyMs: number;
    }>;
    actualGranularity: 'hour' | 'day';  // The granularity actually used
  }

metrics.modelBreakdown
  Input: { projectId?: string, from?: string, to?: string }
  // Defaults: from = 7 days ago, to = now
  Output: {
    models: Array<{
      model: string;
      generationCount: number;
      inputTokens: number;
      outputTokens: number;
      cost: number;
    }>;
  }

metrics.export
  Input: { projectId?: string, from: string, to: string, format: 'jsonl' | 'csv', scope: 'traces' | 'observations' }
  // from/to required because export without explicit range is dangerous
  Output: { jobId: string }  // BullMQ job id
```

### 4.2 REST Export Status

```
GET /api/public/exports/:jobId
```

Returns job status (pending / completed / failed) and MinIO download URL when done.

## 6. Export Pipeline Design

### 6.1 Job Payload

```typescript
interface ExportJob {
  projectId: string;
  from: string;
  to: string;
  format: "jsonl" | "csv";
  scope: "traces" | "observations";
  // Permission check uses projectId match, not apiKeyId.
  // apiKeyId is stored for audit only; session-authenticated exports also work.
  requestedBy: { userId?: string; apiKeyId?: string };
}
```

### 6.2 Worker Flow

1. **Query CH** — Stream rows via `query` + `JSONEachRow` format

   ```typescript
   const resultSet = await ch.query({ query, format: "JSONEachRow" });
   const stream = resultSet.stream();
   for await (const rows of stream) {
     for (const row of rows) {
       // row is a parsed JSON object
       // write to local temp file / transform / stream to S3
     }
   }
   ```

2. **Transform**:
   - `jsonl`: one JSON object per line, Decimal64 fields serialized as strings to avoid precision loss
   - `csv`: explicit headers mapped from CH column names to human-readable labels; Decimal fields as strings
3. **Upload to MinIO/S3**:
   - Key: `exports/{projectId}/{scope}_{from}_{to}_{timestamp}.{format}`
   - Bucket: `constell` (reuse existing MinIO config)
4. **Mark complete** — Return `{ downloadUrl, expiresAt }` from the job handler. BullMQ stores this as `job.returnvalue`.
   - Also write a backup to Redis: `SET export-result:{jobId} <json> EX 86400` (24h TTL).
   - The status endpoint checks `job.returnvalue` first, falls back to Redis. This guards against BullMQ job retention cleanup.\n

### 6.3 REST Status Endpoint

```
GET /api/public/exports/:jobId
```

**Permission check**: The endpoint resolves `projectId` from the caller's session or API key, then verifies the job's `projectId` matches. Returns `404` (not `403`) if the job exists but belongs to a different project — this prevents enumeration attacks.

**Result lookup** (in order):

1. Fetch job from BullMQ queue via `Job.fromId(queue, jobId)`.
2. If `job.returnvalue` exists → `completed`.
3. If job state is `failed` → `failed` with error message.
4. If job exists but no return value → `pending`.
5. Fall back to Redis `GET export-result:{jobId}` for completed jobs that were cleaned from BullMQ.
6. If none found → `404`.

```json
// 200 OK
{
  "status": "completed",
  "downloadUrl": "http://localhost:9090/constell/exports/proj-xxx/traces_2026-01-01_2026-01-31_1234567890.jsonl",
  "expiresAt": "2026-06-01T00:00:00Z"
}

// 202 Accepted
{ "status": "pending", "position": 3 }

// 404 Not Found (job doesn't exist OR belongs to another project)
{ "error": "Export job not found" }

// 500 Internal Server Error
{ "status": "failed", "error": "ClickHouse query timeout" }
```

// packages/shared/prisma/seed.ts

import { seedModelPrices } from "./seed/modelPrices";

const prisma = new PrismaClient();

## 10. Performance & Scale Considerations

| Concern            | Mitigation                                                                                                                              |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------------- |
| CH scan too large  | Enforce max 90-day window in API; CH has TTL on `ingested_at + 90d`                                                                     |
| Export job memory  | Stream CH rows, don't buffer entire result set                                                                                          |
| Concurrent exports | BullMQ concurrency = 2 per worker instance                                                                                              |
| Dashboard load     | No caching in v0.5.0-beta. CH `FINAL` + `project_id` prefix scan is fast enough. Redis caching is a post-v0.5.0 optimization if needed. |
| Decimal precision  | Cast CH `Decimal64` to string in JSON; parse in frontend                                                                                |

## 11. Security

- All metrics endpoints use `authedProcedure` (session or API key)
- `projectId` is always resolved from auth context; user-provided `projectId` is only a fallback for API-key flows
- Export URLs are signed or time-limited (MinIO presigned URLs)
- Export jobs are scoped to `projectId`. The status endpoint resolves the caller's `projectId` from session or API key and verifies the job belongs to that project. Returns `404` (not `403`) for mismatches to prevent job-id enumeration.

- [ ] `metrics.summary` returns accurate KPIs; empty ranges return `0` for all fields (no `NaN`)
- [ ] `metrics.trends` returns daily buckets with cost/latency/token trends; auto-aggregates to day granularity when > 30 buckets
- [ ] `metrics.modelBreakdown` returns per-model usage and cost; excludes empty/unnamed models
- [ ] Dashboard page renders KPI cards, trend bars, and model table; shows empty-state placeholder when no data
- [ ] `metrics.export` enqueues a job via `export-queue`; worker generates `.jsonl` and uploads to MinIO
- [ ] `GET /api/public/exports/:jobId` returns download URL when complete; returns `404` for cross-project access
- [ ] `ModelPrice` seed script populates 6 common models
- [ ] All typechecks pass; all tests pass
- [ ] CHANGELOG updated with v0.5.0-beta entry
