# v0.7.0-beta — Evaluations Design Spec

> **Status**: Draft  
> **Target Release**: v0.7.0-beta  
> **Depends on**: v0.6.0-beta (Scores)  
> **Frozen At**: —

---

## 1. Goal

Add an **automated evaluation pipeline** that runs LLM-as-a-Judge against traces and observations to produce scores. Evaluations close the loop between observability ("what happened") and quality assessment ("was it good?") by removing the manual effort of scoring.

**Exit**: Users can create eval templates, trigger eval runs manually or on new traces, view eval results, and debug judge outputs.

---

## 2. Philosophy: What Constell Evaluations Are (and Aren't)

Constell does not aim to replicate Langfuse's full annotation queue workflow (team-based manual labeling). That remains a future **Annotations** phase (v0.8.0+).

**v0.7.0 scopes evaluations to three things:**

1. **Template-driven judge prompts** — Define what the LLM evaluates (e.g., "Rate helpfulness 1-5") with structured output parsing
2. **One-shot eval runs** — Trigger a run against a time range of traces; worker executes judge calls asynchronously
3. **Result inspection** — View generated scores (source=EVAL) alongside manual scores

**Not in scope:**

- Annotation queues / team-based manual review workflows
- Continuous eval (auto-trigger on every new trace) — post-v0.7.0 with queue hooks
- Multi-model judge ensembles (single judge per eval run only)
- Custom judge code execution (only LLM API judges)
- Evaluation comparison / A/B testing across judge versions

```
┌────────────────────────────────────────────────────────────────────┐
│                           Web (Next.js)                             │

│  │            │  │ router     │  │ Viewer     │                    │

└────────────────────────┼───────────────────────────────────────────┘
                         │
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
  ┌──────────────┐ ┌─────────────┐ ┌──────────────┐
  │ PostgreSQL   │ │ Redis Queue │ │ LLM Provider │
  │ EvalTemplate │ │ eval-queue  │ │ (OpenAI etc) │
  │ EvalRun      │ │             │ │              │
  └──────────────┘ └─────────────┘ └──────────────┘
                         │
                         ▼
                  ┌──────────────┐
                  │ Worker       │
                  │ evalProcessor│
                  └──────┬───────┘
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
       ┌────────┐  ┌──────────┐  ┌──────────┐
       │ PG     │  │ CH       │  │ Score    │
       │ Score  │  │scores_wide│ │ (EVAL)   │
       └────────┘  └──────────┘  └──────────┘
```

**Design Principles:**

- **Eval results are just scores** — They write to the existing `Score` model with `source=EVAL`. No separate eval result table. This reuses all existing score infrastructure (analytics, trace detail display, dashboard).
- **Templates are version-agnostic** — An eval template is a prompt + output parser. Changing a template creates a new template; runs reference templates immutably.
- **Worker handles LLM calls** — The eval processor fetches traces from ClickHouse, calls the LLM API, parses structured output, and writes scores. All LLM calls are in the worker, never the web tier.
- **Idempotent runs** — Re-running the same eval run against the same traces overwrites previous EVAL scores via `ReplacingMergeTree` (same `trace_id` + `name` + newer `event_ts`).

## 4. Data Model

### 4.1 PostgreSQL (Prisma)

**`EvalTemplate`** — Defines a judge prompt and output schema.

```prisma
model EvalTemplate {
  id          String   @id @default(cuid())
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @default(now()) @updatedAt @map("updated_at")
  projectId   String   @map("project_id")
  project     Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)

  name        String
  description String?
  prompt      String   @db.Text

  // Output parsing config
  outputSchema Json?   @map("output_schema") // {"type": "number", "path": "score"} or {"type": "string", "path": "label"}

  // Scoring config
  scoreName   String   @map("score_name")    // e.g. "helpfulness"
  scoreDataType ScoreDataType @map("score_data_type") // NUMERIC, BOOLEAN, CATEGORICAL

  // Judge config
  model       String   @default("gpt-4o-mini") // LLM model to use for judging
  temperature Float?   @default(0.0)

  evalRuns    EvalRun[]

  @@unique([projectId, name])
  @@index([projectId])
  @@map("eval_templates")
}
```

> **Rationale**: `outputSchema` is JSON to support future extensibility (categorical labels, multi-field extraction). For v0.7.0, it only supports extracting a single number/string at a JSON path.

**`EvalRun`** — Records a single execution of an eval template against a trace set.

```prisma
model EvalRun {
  id            String   @id @default(cuid())
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @default(now()) @updatedAt @map("updated_at")
  projectId     String   @map("project_id")
  project       Project  @relation(fields: [projectId], references: [id], onDelete: Cascade)

  templateId    String   @map("template_id")
  template      EvalTemplate @relation(fields: [templateId], references: [id], onDelete: Cascade)

  status        EvalRunStatus @default(PENDING)

  // Run parameters
  from          DateTime? @map("from_date")   // Time range of traces to evaluate
  to            DateTime? @map("to_date")
  traceCount    Int?      @map("trace_count") // Number of traces matched
  processedCount Int?     @map("processed_count")

  // Results summary
  scoreCount    Int?      @map("score_count") // Number of scores generated
  averageScore  Float?    @map("average_score")

  // Error tracking
  errorMessage  String?   @map("error_message") @db.Text

  // Job tracking
  jobId         String?   @map("job_id")      // BullMQ job ID

  @@index([projectId, templateId])
  @@index([status])
  @@map("eval_runs")
}

enum EvalRunStatus {
  PENDING
  RUNNING
  COMPLETED
  FAILED
  CANCELLED
}
```

> **Rationale**: EvalRun is a lightweight status tracker. The actual results are stored as `Score` records with `source=EVAL`. This avoids duplicating score data. The `from`/`to` range lets users re-run the same eval on newer traces.

Add relations to existing models:

- `Project`: add `evalTemplates EvalTemplate[]` and `evalRuns EvalRun[]`

## 5. Worker Pipeline

### 5.1 Eval Processor

```
1. Load EvalTemplate + EvalRun from PG
2. Query ClickHouse for traces in [from, to] range
3. For each trace:
   a. Render prompt (template.prompt + trace data)
   b. Call LLM API (OpenAI/Anthropic via fetch)
   c. Parse structured output (JSON path extraction)
   d. Create Score (source=EVAL) via Prisma
   e. Write Score to CH scores_wide
4. Update EvalRun status + summary
```

**Prompt rendering**: Simple variable substitution. The template prompt can include `{trace.input}`, `{trace.output}`, `{observation.input}`, `{observation.output}`, `{trace.name}`, etc.

**LLM API**: Use native `fetch` with the project's LLM API key (stored in env). Support OpenAI-compatible APIs only for v0.7.0.

**Output parsing**:

```json
// outputSchema examples:
{"type": "number", "path": "score"}        // Extracts result.score as numeric
{"type": "boolean", "path": "passed"}      // Extracts result.passed as 0/1
{"type": "string", "path": "label"}        // Extracts result.label as categorical
```

### 5.2 Error Handling

- **LLM API failure**: Log error, skip trace, continue with others. Increment `processedCount`.
- **Parse failure**: If output doesn't match schema, skip trace.
- **Partial failure**: Run status = COMPLETED if ≥1 trace succeeded, with `errorMessage` listing failures.
- **Total failure**: Run status = FAILED if 0 traces succeeded.

## 6. API Design

### 6.1 tRPC `evals` Router

```typescript
evals.templateList
  Input: { projectId?: string }
  Output: Array<{ id, name, description, scoreName, scoreDataType, model }>

evals.templateCreate
  Input: {
    projectId?: string;
    name: string;
    description?: string;
    prompt: string;
    outputSchema?: Json;
    scoreName: string;
    scoreDataType: "NUMERIC" | "BOOLEAN" | "CATEGORICAL";
    model?: string;
    temperature?: number;
  }
  Output: { id: string }

evals.templateDelete
  Input: { projectId?: string; id: string }
  // Rejected if any EvalRun references this template
  Output: { success: boolean }

evals.runList
  Input: { projectId?: string; templateId?: string; limit?: number; offset?: number }
  Output: { runs: Array<{ id, templateName, status, traceCount, scoreCount, averageScore, createdAt }>, total: number }

evals.runCreate
  Input: {
    projectId?: string;
    templateId: string;
    from?: string; // ISO date
    to?: string;   // ISO date
  }
  // Enqueues eval job, returns immediately with runId
  Output: { id: string, status: "PENDING" }

evals.runDetail
  Input: { projectId?: string; id: string }
  Output: {
    id, templateId, templateName, status,
    from, to, traceCount, processedCount, scoreCount, averageScore,
    errorMessage, createdAt, updatedAt
  }

evals.runCancel
  Input: { projectId?: string; id: string }
  // Only works if status is PENDING or RUNNING
  Output: { success: boolean }
```

### 6.2 Public REST API

No public REST API for evals in v0.7.0. Eval triggers are UI-only. Public API for evals is deferred to post-v0.7.0.

## 8. Security

- All eval endpoints use `authedProcedure` only (no public REST API in v0.7.0)
- `projectId` resolved from auth context; cross-project access returns `404`
- LLM API keys are stored in server-side env vars only (`OPENAI_API_KEY`, `ANTHROPIC_API_KEY`)
- Eval prompts are sandboxed — only trace/observation data is injected, no arbitrary code execution
- EVAL scores remain immutable via UI/API (enforced by existing tRPC/REST logic)

## 9. Performance & Scale

| Concern             | Mitigation                                                                                  |
| ------------------- | ------------------------------------------------------------------------------------------- |
| LLM API rate limits | Worker processes eval jobs sequentially within a run; BullMQ concurrency = 1 per eval queue |
| Long-running evals  | Runs against large trace sets are chunked; each trace is an independent LLM call            |
| Cost control        | Default model is `gpt-4o-mini`; template-level model override requires explicit choice      |
| PG write volume     | Scores are batched in groups of 10 during eval processing                                   |
| CH write volume     | Scores are written to CH via existing `ClickHouseBatchWriter` after PG insert               |

## 11. Post-v0.7.0 Ideas (Not in Scope)

- **Continuous eval** — Auto-trigger eval on new trace ingestion via ingestion pipeline hooks
- **Annotation queues** — Team-based manual review workflow with assignment
- **Multi-judge ensemble** — Run multiple judges and aggregate results
- **Custom judge code** — Allow users to upload Python/JS judge functions
- **Eval comparison** — Compare results across different judge versions or models
- **Public eval API** — Trigger eval runs via REST API
